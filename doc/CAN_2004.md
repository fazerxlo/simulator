
## Draft: CAN 2004 messages used by `fazerxlo/simulator`

The simulator sends periodic CAN frames on a **125 kbit/s socketcan bus (`can0`)** via the generic sender loop in `can_runner.py`. Modules enabled by default in `config.yml` are `bsi-base`, `bsi-trip`, `bsi-log`, and `clim`. Each module registers message producers with a schedule in milliseconds. The documentation below reflects the frames explicitly generated by those modules.  

### 1. BSI / Base messages

#### `0x036` — BSI commands
Period: **50 ms**

Produced by `BSI_base.can_commandes()`.

| Byte | Meaning |
|---|---|
| 0 | Constant `0x0E` |
| 1 | Constant `0x00` |
| 2 | Bit 7 = economy mode |
| 3 | Bit 5 = dash lights, Bit 4 = dark mode, low bits = luminosity value |
| 4 | Power mode / ignition state |
| 5 | Constant `0x80` |
| 6 | Constant `0x00` |
| 7 | Constant `0xA0` |

Known power mode values:
- `0x01` = ignition on
- `0x02` = ignition off
- `0x03` = ignition wakeup

Example:
```text
0x036  0E 00 00 0A 02 80 00 A0
```

#### `0x0F6` — Slow vehicle/body status (BSI_SLOW_DATA)
Period: **100 ms** (simulator); **500 ms** on real bus (PSA-RE)

Produced by `BSI_base.can_slow()`.

| Byte | Meaning | Notes |
|---|---|---|
| 0 | Status byte: CONFIG_MODE (bits 7-6), MAIN_STATUS (bits 4-3), GEN_STATUS (bit 2), POWERTRAIN_STATUS (bits 1-0) | Real bus = `0x88`; simulator sends `0x08` |
| 1 | Coolant temperature: `raw − 40 °C` | |
| 2-4 | **Odometer** (uint24): `raw × 0.1 km`, invalid = `0xFFFFFF` | Simulator sends constants `0x00 0x1F 0x00` |
| 5 | External temperature: `raw × 0.5 − 40 °C`, invalid = `0xFF` | |
| 6 | External temperature (filtered): same formula | damped/smoothed copy |
| 7 | Bit 7 = REVERSE_STATUS; bit 6 = FRONT_WIPERS_STATUS; bits 5-4 = WHEEL_POSITION; bit 3 = lamp test; bits 1-0 = BLINKERS_STATUS (0=none, 1=right, 2=left, 3=both) | |

Notes:
- External temperature range is limited in UI from **-40°C to 85°C**.
- Reverse flag (bit 7 of byte 7) enables parking sensor frame `0x0E1`.
- Signal positions verified against [PSA-RE AEE2004 LS.CONF](https://github.com/prototux/PSA-RE). See `doc/PSA_RE_comparison.md`.

#### `0x0B6` — Fast drivetrain data
Period: **50 ms**

Produced by `BSI_base.can_fast()`.

| Byte | Meaning |
|---|---|
| 0 | RPM high byte |
| 1 | RPM low byte |
| 2 | Speed high byte |
| 3 | Speed low byte |
| 4 | Constant `0x00` |
| 5 | Constant `0x00` |
| 6 | Constant `0x00` |
| 7 | Constant `0x00` |

Encoding:
- RPM raw = displayed RPM << 3
- Speed = displayed speed × 100

#### `0x0E1` — Parking assist / reverse-related frame
Period: **100 ms when reverse is active**
Produced by `BSI_base.parktronic()`.

Sent only when reverse is enabled.

Payload:
```text
24 D0 CC 11 5C FE C2
```

If reverse is off, the frame is not sent.

#### `0x2B6` — VIN VIS portion
Period: **100 ms**

Produced by `BSI_base.can_vin_vis()`.

Payload is fixed ASCII:
```text
32 31 37 31 35 33 38 33
```
ASCII: `21715383`

#### `0x336` — VIN WMI portion
Period: **100 ms**

Produced by `BSI_base.can_vin_wmi()`.

Payload:
```text
56 46 33
```
ASCII: `VF3`

#### `0x3B6` — VIN VDS portion
Period: **100 ms**

Produced by `BSI_base.can_vin_vds()`.

Payload:
```text
36 4A 52 48 52 48
```
ASCII: `6JRHRH`

Combined simulated VIN parts:
- WMI: `VF3`
- VDS: `6JRHRH`
- VIS: `21715383`

#### `0x161` — Oil temperature / fuel level / oil level (BSI_GAUGES)
Period: **500 ms** (simulator); **500 ms** on real bus (PSA-RE)

Produced by `Msg161` in `can_messages.py`.

| Byte | Meaning | Notes |
|---|---|---|
| 0 | OIL_LEVEL_RESTART (bit 7) | restart flag; simulator sends `0x00` |
| 1 | (unused) | |
| 2 | OIL_TEMPERATURE source byte; received raw bus data decodes as `raw − 40 °C`, while simulator transmission uses a workbench-derived linear conversion for closer cluster alignment | |
| 3 | FUEL_LEVEL: 0–100 %, invalid = `0xFF` | |
| 4-5 | (unused) | simulator sends `0xFF` |
| 6 | **OIL_LEVEL**: 0–100 %, invalid = `0xFF` | new signal from PSA-RE; simulator sends `0xFF` |

Note: PSA-RE defines this as a 7-byte frame, and the current simulator message implementation follows that length. Signal positions verified against [PSA-RE AEE2004 LS.CONF](https://github.com/prototux/PSA-RE).

### 2. BSI / Trip computer messages

#### `0x221` — Instant trip data
Period: **100 ms**

Produced by `BSI_trip.can_inst()`.

| Byte | Meaning |
|---|---|
| 0 | Bit 7 = hide fuel, Bit 6 = hide distance, Bit 3 = right command, Bit 0 = left command |
| 1 | Instant fuel consumption high byte |
| 2 | Instant fuel consumption low byte |
| 3 | Autonomy high byte |
| 4 | Autonomy low byte |
| 5 | Distance to destination high byte |
| 6 | Distance to destination low byte |

Encoding:
- Fuel consumption = UI value × 10
- Autonomy = integer km
- Distance = UI value × 10

#### `0x2A1` — Trip 1 history
Period: **100 ms**

Produced by `BSI_trip.can_trip1()`.

| Byte | Meaning |
|---|---|
| 0 | Average speed |
| 1 | Distance high byte |
| 2 | Distance low byte |
| 3 | Average fuel high byte |
| 4 | Average fuel low byte |
| 5 | Constant `0x00` |
| 6 | Constant `0x00` |

Notes:
- Distance is integer km.
- Fuel = displayed value × 10.
- The two trailing zero bytes are explicitly required by the author comment, otherwise the display shows `"--"`.

#### `0x261` — Trip 2 history
Period: **100 ms**

Produced by `BSI_trip.can_trip2()`.

Same layout as `0x2A1`, but for trip memory 2.

### 3. BSI / Log or popup message frame

#### `0x1A1` — Dashboard textual/status message trigger
Period: **100 ms**

Produced by `BSI_log.can_message()`.

| Byte | Meaning |
|---|---|
| 0 | Message flag |
| 1 | Message ID |
| 2 | `0xF0` when a message is active, else `0x00` |
| 3 | Constant `0x00` |
| 4 | Constant `0x00` |
| 5 | Constant `0x00` |
| 6 | Constant `0x00` |
| 7 | Constant `0x00` |

Observed flag behavior from code:
- `0xFF` = idle / no popup
- `0x80` = message just triggered
- `0x7F` = reset/clear stage before returning to idle

Selected message ID values from `modules/bsi-log/messages.py`:
- `0x00` = Diagnostic OK
- `0x01` = STOP: engine temperature fault
- `0x31` = Automatic headlights enabled
- `0x32` = Automatic headlights disabled
- `0x33` = Auto door locking enabled
- `0x34` = Auto door locking disabled
- `0x61` = Parking brake applied
- `0x62` = Parking brake released
- `0x88` = Parking assist faulty
- `0xD8` = Risk of ice
- `0xD9` = Handbrake forgotten
- `0xDE` = Driver door open
- `0xDF` = Washer fluid low
- `0xE0` = Low fuel level
- `0xF0` = Diagnostic in progress
- `0xF1` = Diagnostic complete

There are many more IDs in the file; if you want, I can format the full list into a proper table.

### 4. Climate control messages

#### `0x1D0` — Climate panel state
Period: **500 ms**

Produced by `Clim.can_clim_panel()`.

When the climate module is not loaded or ignition is off, the BSI idle frame
`08 00 00 00 00 0B 0B 00` is sent.

When climate is active:

| Byte | Meaning |
|---|---|
| 0 | `0x08` base constant; `0x19` when front demist is active |
| 1 | Constant `0x00` |
| 2 | Fan speed raw nibble (0x00–0x07 = levels 1–8, 0x0F = off) |
| 3 | High nibble = left zone air distribution, low nibble = right zone |
| 4 | Bit 5 = recirculation, Bit 4 = front demist |
| 5 | Left temperature index |
| 6 | Right temperature index |

Temperature index mapping:
- `0` = LO
- `1` = 14°C
- `2` = 15°C
- `3` = 16°C
- `4` = 17°C
- `5` = 18°C
- `6` = 18.5°C
- `7` = 19°C
- `8` = 19.5°C
- `9` = 20°C
- `10` = 20.5°C
- `11` = 21°C
- `12` = 21.5°C
- `13` = 22°C
- `14` = 22.5°C
- `15` = 23°C
- `16` = 23.5°C
- `17` = 24°C
- `18` = 25°C
- `19` = 26°C
- `20` = 27°C
- `21` = 28°C
- `22` = HI

#### `0x12D` — Climate command frame
Period: **500 ms**

Suppressed when ignition is off.  Workbench captures show a constant payload
throughout the entire climate operating scenario:

```text
00 32 32 00 00 00 98 80
```

Bytes 1–2 (`0x32 0x32`) and bytes 6–7 (`0x98 0x80`) appear to be fixed
controller-identity or capability fields.

#### `0x1E3` — Climate display / EMF state
Period: **200 ms**

When the climate module is not loaded or ignition is off, the BSI standby
frame is sent (`0x1C 0x30 0x0B 0x0B 0x00 0x00 0x00 0x00` when ignition on,
or `0x1C 0x40 ...` when ignition off).

When climate is active:

| Byte | Meaning |
|---|---|
| 0 | `0x14` constant \| Bit 3 = auto \| Bit 0 = dual |
| 1 | `0x30` constant \| Bit 7 = front demist |
| 2 | `clim.bits` \| Left temperature index (bits 0–4) |
| 3 | Right temperature index |
| 4 | Left zone air distribution in high nibble |
| 5 | Right zone air distribution in high nibble |
| 6 | Fan speed raw nibble (0x00–0x07 = levels 1–8, 0x0F = off) |
| 7 | Constant `0x00` |

Byte 0 encodes: `0x14 | (auto << 3) | dual`.
Byte 1 encodes: `0x30 | (unfrost_front << 7)`.

## Message schedule summary

| CAN ID | Source module | Period | Purpose |
|---|---|---:|---|
| `0x036` | bsi-base | 50 ms | BSI commands / ignition / lighting |
| `0x0B6` | bsi-base | 50 ms | Fast RPM/speed |
| `0x0F6` | bsi-base | 100 ms | Slow body/temperature/reverse |
| `0x0E1` | bsi-base | 100 ms* | Parking assist in reverse |
| `0x161` | bsi-base | 100 ms | Oil temp + fuel level |
| `0x2B6` | bsi-base | 100 ms | VIN VIS |
| `0x336` | bsi-base | 100 ms | VIN WMI |
| `0x3B6` | bsi-base | 100 ms | VIN VDS |
| `0x221` | bsi-trip | 100 ms | Instant trip data |
| `0x2A1` | bsi-trip | 100 ms | Trip 1 history |
| `0x261` | bsi-trip | 100 ms | Trip 2 history |
| `0x1A1` | bsi-log | 100 ms | Text/status popup messages |
| `0x1D0` | clim | 500 ms | Climate panel state |
| `0x12D` | clim | 500 ms | Climate command frame |
| `0x1E3` | clim | 200 ms | Climate EMF/display state |

\* only sent when reverse is active.
