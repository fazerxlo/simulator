Here’s the **best-effort CAN2004 documentation for frame `0x120`** from this repo, with focus on **tire statuses**.

The sketch treats `0x120` as an **8-byte alerts journal / diagnostic frame** and decodes it into **three blocks**, selected by the top two bits of `data[0]`:
- **Block 1:** `data[0].bit7=0`, `bit6=1`
- **Block 2:** `data[0].bit7=1`, `bit6=0`
- **Block 3:** `data[0].bit7=1`, `bit6=1`

Source decoding is implemented here:

## Frame structure summary

| Byte | Meaning |
|---|---|
| `data[0]` | Block selector, using bits 7 and 6 |
| `data[1]..data[7]` | Bitfields containing alert/status flags for that block |

## Block selector

| `data[0] bit7` | `data[0] bit6` | Block |
|---|---:|---|
| 0 | 1 | Block 1 |
| 1 | 0 | Block 2 |
| 1 | 1 | Block 3 |

## 1. Block 2: Puncture wheel(s)

This is the clearest tire mapping.

```c++ name=arduino-psa-comfort-can-adapter.ino url=https://github.com/ludwig-v/arduino-psa-comfort-can-adapter/blob/b54faf80144a8132a716be26f55bac4186b0f016/arduino-psa-comfort-can-adapter/arduino-psa-comfort-can-adapter.ino#L1017-L1023
notificationParameters = 0x00;
bitWrite(notificationParameters, 7, bitRead(canMsgRcv.data[1], 4)); // Front left tyre
bitWrite(notificationParameters, 6, bitRead(canMsgRcv.data[1], 3)); // Front right tyre
bitWrite(notificationParameters, 5, bitRead(canMsgRcv.data[1], 2)); // Rear right tyre
bitWrite(notificationParameters, 4, bitRead(canMsgRcv.data[1], 1)); // Rear left tyre
sendPOPup((bitRead(canMsgRcv.data[1], 4) || bitRead(canMsgRcv.data[1], 3) || bitRead(canMsgRcv.data[1], 2) || bitRead(canMsgRcv.data[1], 1)), 13, 6, notificationParameters);
```

### Meaning
When in **Block 2**:
- `data[1].bit4` = **Front left tyre puncture**
- `data[1].bit3` = **Front right tyre puncture**
- `data[1].bit2` = **Rear right tyre puncture**
- `data[1].bit1` = **Rear left tyre puncture**

If any of these are set, the sketch emits popup ID `13` with priority `6`.
